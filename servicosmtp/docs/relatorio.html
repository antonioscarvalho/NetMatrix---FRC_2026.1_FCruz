1. O que foi implementado

Servidor de correio eletrônico institucional para a intranet NetMatrix, responsável pelo recebimento, triagem e entrega local de mensagens via protocolo SMTP (Simple Mail Transfer Protocol), além de preparar a infraestrutura para a leitura de mensagens via protocolo POP3.

O roteiro original (UnB/FCTE) assume uma instalação padrão do Postfix no Ubuntu/Debian. O sistema foi configurado para atuar como o agente de transferência de mensagens (MTA) oficial do domínio da empresa, operando de forma integrada ao ambiente de subsistema Linux (WSL2).

2. Topologia de teste

Como o protocolo SMTP corporativo trafega sobre conexões TCP fim-a-fim, os testes iniciais foram consolidados dentro do ambiente isolado do terminal Linux, garantindo que o Postfix fizesse a escuta correta em todas as interfaces lógicas:

[ Usuário: antonio ] --(Injeção Local: comando mail)--> [ MDA: cleanup/local ] | (/var/mail/luana) v [ Usuário: luana ] <--(Acesso Remoto: telnet 110)-- [ Spool: mbox ]
Nota técnica: Durante o primeiro provisionamento no WSL, o Postfix rejeitou as entregas gerando mensagens de bounce (Host or domain name not found) devido à falta de sincronia entre o nome local do Host (DESKTOP-KGELT00.localdomain) e a identidade real do domínio. A resolução exigiu o alinhamento forçado do arquivo /etc/mailname e a amarração explícita no arquivo global de parâmetros.

3. Configuração do servidor (main.cf)

Abaixo encontra-se o arquivo de configuração global /etc/postfix/main.cf limpo, contendo apenas as diretivas operacionais ativas que corrigiram os problemas de transporte de correio:

# Identidade do Servidor e Domínio
myhostname = mail.netmatrix.com.br
myorigin = /etc/mailname

# Banner de saudação SMTP (Conforme exigido pelas RFCs)
smtpd_banner = $myhostname ESMTP $mail_name (Ubuntu)

# Escuta de Redes e Protocolos
inet_interfaces = all
inet_protocols = all

# Definição de Destino Final (Autoridade sobre o domínio)
mydestination = netmatrix.com.br, mail.netmatrix.com.br, localhost.localdomain, localhost

# Redes Confiáveis (Permitidas a enviar sem autenticação prévia)
mynetworks = 127.0.0.0/8, 172.0.0.0/8

# Parâmetros de Armazenamento Local (Sem cotas)
mailbox_size_limit = 0
recipient_delimiter = +
biff = no

5. Observação sobre integração ao vivo

Diferente do serviço de DHCP (que fica restrito à camada 2 por depender de broadcasts), o tráfego SMTP/POP3 opera inteiramente na camada de aplicação sobre conexões Unicast TCP estáveis. Isso significa que, assim que a Luana finalizar a tabela de encaminhamento no servidor DNS do grupo, a integração entre as máquinas físicas do laboratório poderá ocorrer remotamente através da VPN do Tailscale sem barreiras de roteamento.

6. Respostas às questões do roteiro

1) Descreva as principais modificações em /etc/postfix/main.cf e master.cf.
No arquivo main.cf, as modificações definem os parâmetros globais do servidor (hostname, destinos e redes confiáveis). No arquivo master.cf, ativam-se os daemons essenciais em segundo plano, como o smtpd para escutar conexões externas na porta 25 e o processo local para gravação física das caixas.
2) Qual a diferença entre mbox e maildir?
O formato mbox concatena todas as mensagens recebidas de um usuário em um único arquivo de texto plano localizado em /var/mail/usuario. O formato maildir cria uma estrutura organizada de subpastas e armazena cada e-mail individual em um arquivo isolado.
3) Para que servem os esquemas SASL/TLS?
O SASL provê uma camada para autenticação segura de credenciais do usuário impedindo ações de Open Relay. O TLS criptografa toda a sessão de dados de ponta a ponta, evitando o monitoramento de informações sensíveis por sniffers na rede.
4) Qual a diferença entre RFC822 e MIME types?
A RFC822 define os cabeçalhos básicos e corpos textuais puramente em ASCII de 7 bits. O padrão MIME estendeu essa especificação, possibilitando o transporte de formatações complexas em HTML, múltiplos anexos binários e codificação de caracteres internacionais (UTF-8).
